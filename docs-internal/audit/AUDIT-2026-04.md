# Kimberlite Independent Compliance-Gap Audit — Fourth Assessment

**Audit Date:** April 2026
**Target Commit:** `89d3bd6` (fix(tenant): scope projection-table catalog by tenant_id)
**Scope:** All 35 workspace crates (kimberlite v0.5.0-pre / post-AUDIT-2026-03)
**Focus Areas:** Claim-reality gaps surfaced by the April 2026 projection-table tenant-isolation bug — places where verification-surface names promise more than the code enforces, where tests/proofs assert the wrong invariant, and where implicit scope assumptions go unasserted.
**Classification:** Internal — Directed Compliance-Claim Review
**Prior Audits:**
- AUDIT-2026-01 (January 2026, v0.9.0) — 21 findings
- AUDIT-2026-02 (February 2026, v0.9.1) — 4 new findings
- AUDIT-2026-03 (February 2026, v0.9.2) — 16 findings, principally Byzantine coverage

---

## Executive Summary

Commit `89d3bd6` fixed a PHI-grade tenant-isolation bug in the SQL catalog — `CREATE TABLE` was globally scoped across tenants. The bug shipped despite 32 Kani proofs, 19 invariant checkers, 46 VOPR scenarios (including one literally named `multi_tenant_isolation`), a 525-line `trace_alignment.rs` mapping TLA theorems to Rust line ranges, and 2800+ tests. Root cause: **the verification harness and the code silently disagreed about what was being enforced, and nothing made the disagreement loud**.

This audit hunts every other place the same class of failure could still exist. We examined (a) all 8 `kani_proofs.rs` files, (b) the full `trace_alignment.rs`, (c) all VOPR scenarios and invariant checkers, (d) the workload generator, (e) the five most load-bearing SDK consumer flows, and (f) compliance documentation claims. Each finding was verified directly against source before inclusion; two speculative agent findings were dropped after verification.

**Overall Risk Rating: HIGH.**

The fix itself ships with the same class of gap it was intended to close: the new `verify_catalog_isolation` checker is defined but never called (F-2), the trace-alignment entries for HIPAA §164.312(a)(1) and GDPR still point at the `(200, 350)` range the commit identified as wrong (F-5), and the scenario named as the TenantIsolationTheorem's evidence has a workload generator that cannot express a `CREATE TABLE` command (F-3). The compliance story most likely to attract a regulator — **Right to Erasure** — is a bookkeeping ledger that deletes nothing (F-1, F-7).

## Methodology

1. **Class-of-bug reconstruction.** We reverse-engineered the five failure modes the projection-table bug exhibited (mis-scoped line ranges, inverted test assertion, missing workload variants, checker keyed on wrong surface, implicit invariant inheritance) and treated them as a pattern-match template for the rest of the codebase.
2. **Parallel structured exploration.** Three `Explore` subagents covered (a) Kani proofs + `trace_alignment.rs`, (b) VOPR scenarios + invariant checkers + workload generator, (c) docs/SDK claims + five consumer walks.
3. **Direct verification.** Every load-bearing agent claim was re-verified against source by the audit author (grep, read, call-site analysis) before inclusion. Two agent findings were dropped (`PreparedQuery` RBAC replay — not wire-reachable; DELETE row-visibility — false for the PK path).
4. **Read-only.** No code was modified during the audit.

## Critical Severity Findings

### C-1: Right-to-Erasure performs no erasure

**CWE:** CWE-221 (Information Loss or Omission), CWE-778 (Insufficient Logging)

**Risk:** CRITICAL — GDPR Art 17 compliance failure

**Description**

`docs/compliance/certification-package.md:290,564` advertises *"Cryptographic erasure (delete tenant DEK)"* and *"Article 17 Right to Erasure: Cryptographic erasure? ✅"*. The implementation performs no erasure.

**Evidence**

```rust
// crates/kimberlite-compliance/src/erasure.rs:327-361
pub fn mark_stream_erased(
    &mut self,
    request_id: Uuid,
    _stream_id: StreamId,   // underscore: unused
    records: u64,
) -> Result<()> {
    // ... bookkeeping only ...
    request.records_erased += records;
    request.status = ErasureStatus::InProgress {
        streams_remaining: streams_remaining - 1,
    };
    Ok(())
}
```

- The `_stream_id` parameter is prefixed with `_` — it is never read.
- No call into `kimberlite-kernel::Command::Delete`.
- No call into `kimberlite-crypto` for key shredding.
- No interaction with stream storage or the projection store.

Then `complete_erasure` (`:372`) computes *"cryptographic erasure proof"* as `SHA-256(request_id || subject_id || records_erased)` — binding only to the self-reported count, not to any act of erasure.

**Attack scenario**

1. Operator with compliance role receives a subject's Article-17 request.
2. They issue `create_erasure_request` and `mark_stream_erased(request_id, StreamId(arbitrary), 10_000)`.
3. `complete_erasure` returns a signed `ErasureAuditRecord` with `erasure_proof_hex`.
4. No data was touched. Regulator receives an attestation of erasure for a file that still contains PHI.

**Remediation**

**Priority: P0 (Pre-Production Blocker)**

1. `mark_stream_erased` must invoke either: (a) `Command::Delete` for every row in `stream_id` belonging to the erasure subject, or (b) key-shredding the stream's DEK via `kimberlite-crypto`, committing to the ciphertext-unrecoverable guarantee the docs claim.
2. The erasure proof should be `HMAC(server-key, stream_id || prev_ciphertext_hash || timestamp)` — binding to the cryptographic act, not the count.
3. Add a Kani proof `verify_complete_erasure_implies_stream_unreadable` that constructs a stream, completes erasure, and asserts `read_events` returns no subject data.
4. Add an integration test in `kimberlite-compliance::tests` that fails today (appends data → runs erasure → re-reads → asserts original bytes present), inverting it after remediation.

### C-2: `verify_catalog_isolation` defined but never called

**CWE:** CWE-754 (Improper Check for Unusual or Exceptional Conditions)

**Risk:** CRITICAL — exact bug-class the fix purports to remediate

**Description**

Commit `89d3bd6` adds a `verify_catalog_isolation` method on `TenantIsolationChecker` and justifies itself with *"so VOPR watches the catalog, not just stream rows."* The method is dead code.

**Evidence**

```
$ rg verify_catalog_isolation
crates/kimberlite-sim/src/query_invariants.rs:611:    pub fn verify_catalog_isolation(
```

One match. The definition. No `.verify_catalog_isolation(...)` call sites anywhere in `crates/kimberlite-sim`, `crates/kimberlite-sim/src/bin/vopr.rs`, scenarios, canaries, or integration tests. The checker's `set_tenant` seed method is also never called.

**Attack scenario**

A future refactor regresses `State::table_name_index` to a global `HashMap<String, _>`. Production `ensure_tenant_owns_table` (`kernel.rs:449, 571`) still panics on mismatch (good), but VOPR reports green and CI goes green. The simulation canary loses the very property the commit adds.

**Remediation**

**Priority: P0**

1. Add a call site in the VOPR command-application hook that invokes `checker.verify_catalog_isolation(table.tenant_id, cmd.tenant_id)` for every `CreateTable`, `CreateIndex`, `DropTable`, `Insert`, `Update`, `Delete` command.
2. Seed the checker via `set_tenant()` at scenario init for each tenant.
3. Add a canary mutation (`sim/canary.rs`) that re-introduces global name-indexing; assert the invariant fires. If the canary passes the invariant, the wiring is wrong.

### C-3: `multi_tenant_isolation` workload cannot issue DDL or DML

**CWE:** CWE-1053 (Missing Documentation for Design)

**Risk:** CRITICAL — compliance-mapped scenario has wrong workload

**Description**

`trace_alignment.rs:166` and `:290` name `multi_tenant_isolation` as the VOPR scenario proving `TenantIsolationTheorem` and `HIPAA_164_312_a_1_TechnicalAccessControl`. The scenario's workload generator cannot express a CREATE TABLE, an INSERT, or a DELETE.

**Evidence**

```rust
// crates/kimberlite-sim/src/workload_generator.rs:102-123
pub enum OpType {
    Read,
    Write,
    ReadModifyWrite,
    Scan { limit: usize },
    BeginTx,
    CommitTx,
    RollbackTx,
}
```

Seven variants. The kernel `Command` enum has nine (`AppendBatch`, `CreateStream`, `CreateStreamWithAutoId`, `CreateTable`, `DropTable`, `CreateIndex`, `Insert`, `Update`, `Delete`). The workload abstraction cannot even translate to DDL/DML; no `OpType → Command` shim exists.

**Impact**

Every DDL/DML arm of `kernel::apply_committed` — including every per-tenant enforcement the `89d3bd6` fix adds — is invisible to the scenario cited as its evidence. The original projection-table bug was invisible because of this, and the condition is unchanged.

**Remediation**

**Priority: P0**

1. Extend `OpType` with `CreateTable { tenant_id, name, columns }`, `DropTable`, `CreateIndex`, `Insert { tenant_id, table_id, row }`, `Update`, `Delete`.
2. Add a translator in the scenario driver: `OpType → kimberlite_kernel::Command`.
3. Add a `catalog_stress` scenario that issues mixed DDL/DML across 5+ tenants at different phases (pre-commit, under fault injection, during view change).
4. Land (C-2) concurrently — the checker and the workload are a matched pair.

### C-4: `mark_stream_erased` ignores its `stream_id`; erasure ledger is forgeable

**CWE:** CWE-345 (Insufficient Verification of Data Authenticity)

**Risk:** CRITICAL — independent of C-1 (even if erasure were real, the ledger would be wrong)

**Description**

`mark_stream_erased` receives a `stream_id` but never cross-checks it against the request's `affected_streams` set or against the subject's actual streams. A caller can pass arbitrary `(stream_id, records)` and accumulate attested erasure counts for streams that were never part of the subject's data footprint.

**Evidence**

```rust
// crates/kimberlite-compliance/src/erasure.rs:327-361 (abbrev.)
pub fn mark_stream_erased(
    &mut self,
    request_id: Uuid,
    _stream_id: StreamId,   // not validated against request.affected_streams
    records: u64,
) -> Result<()> {
    let request = self.find_pending_mut(request_id)?;
    // ... no assert!(request.affected_streams.contains(&stream_id)) ...
    // ... no check that stream_id's tenant matches the subject ...
    request.records_erased += records;
}
```

The `request.affected_streams: Vec<StreamId>` field is populated at request-creation time but is never consulted by `mark_stream_erased`.

**Attack scenario**

Compliance SDK caller fabricates erasure of 10^9 records across arbitrary stream IDs. `complete_erasure` returns a signed attestation referencing the inflated count. A subsequent regulator query sees "subject X had 10^9 records erased" with cryptographic backing.

**Remediation**

**Priority: P0** (concurrent with C-1)

1. `assert!(request.affected_streams.contains(&stream_id))` as a production (not `debug_`) assert per `docs/ASSERTIONS.md`.
2. Cross-check `stream_id.tenant_id() == request.subject.tenant_id()` (or equivalent subject-tenant binding).
3. Cap `records` at the actual stream length at erasure start; reject if exceeded.
4. Add a Kani proof `verify_mark_stream_rejects_foreign_stream`.

### H-2: `ComplianceAuditLog` is volatile RAM with no hash chain; docs advertise "immutable, tamper-evident, hash-chained"

**CWE:** CWE-778 (Insufficient Logging), CWE-117 (Improper Output Neutralization for Logs)

**Risk:** HIGH — regulatory representation failure

**Description**

Three doc locations advertise the compliance audit log as persistent, hash-chained, and tamper-evident. The implementation is `Vec<ComplianceAuditEvent>` in process memory.

**Evidence — the claim**

```
docs/operating/security.md:134                   — "Immutable append-only audit log"
docs/operating/production-deployment.md:44       — "Immutable audit logs: Hash-chained, tamper-evident"
docs/operating/production-deployment.md:233      — audit_log_immutable = true  # Tamper-evident hash chain
docs/compliance/certification-package.md:362     — "provably immutable (hash-chained)"
crates/kimberlite-compliance/src/audit.rs:335    — "structural guarantee — the API provides no mutation or deletion methods"
```

**Evidence — the code**

```rust
// crates/kimberlite-compliance/src/audit.rs:337-340
#[derive(Debug, Default)]
pub struct ComplianceAuditLog {
    events: Vec<ComplianceAuditEvent>,
}
```

No persistence. No `prev_hash` field on `ComplianceAuditEvent` (definition at `:71-237`). No fsync. No recovery. Server restart = the log is gone.

The "API offers no delete" docstring is true for the in-memory API; it does not constitute tamper-evidence because any process-level actor (crash, OOM kill, kubelet rollout, planned maintenance) clears the log. The "provably immutable" marketing phrase at `certification-package.md:362` cannot be formally backed — there is no cryptographic linkage between events to prove properties about.

**Remediation**

**Priority: P1**

1. Persist audit events to a dedicated append-only storage segment (reuse `kimberlite-storage`) with fsync on append.
2. Add `prev_hash: Hash` to `ComplianceAuditEvent`; compute on append.
3. Add Kani proof `verify_audit_chain_tamper_detected` and TLA+ theorem `ComplianceAuditChainIntegrity`.
4. Integration test: append N events → drop engine → reload → assert all events and chain valid.

### H-3: Liveness invariants (`EventualCommitChecker`, `EventualProgressChecker`) are never instantiated

**CWE:** CWE-833 (Deadlock)

**Risk:** HIGH — liveness is a headline VOPR property, unverified

**Description**

CLAUDE.md advertises *"19 invariant checkers validating consensus safety, storage integrity, offset monotonicity, and MVCC correctness"*, and the traceability matrix cites liveness invariants for scenarios like `gray_failures`, `view_change_recovery`, `protocol_attacks::byzantine_attacks`. Both liveness checkers exist only as source definitions.

**Evidence**

```
$ rg EventualCommitChecker|EventualProgressChecker --type=rust
crates/kimberlite-sim/src/liveness_invariants.rs  # definitions only
# No construction, no .tick(), no .check(), no .on_prepare / .on_commit calls elsewhere.
```

**Impact**

A prepared operation that never commits despite a durable quorum — a classical liveness bug under gray failure — is invisible to VOPR. View-change livelock, commit-number starvation, and repair-starvation scenarios have no observer.

**Remediation**

**Priority: P1**

1. Instantiate both checkers in the VOPR scenario driver.
2. Wire `on_prepare` / `on_commit` to the message bus, `tick()` to the simulated clock, `set_quorum_live()` to the fault model.
3. Add a canary mutation that drops all `PrepareOk` for a prepared op; assert the checker fires.

### H-4: "Cryptographic erasure proof" binds to self-reported count, not to any cryptographic act

**CWE:** CWE-347 (Improper Verification of Cryptographic Signature)

**Risk:** HIGH — separate from C-1 (even if erasure were real, the proof artifact is wrong shape)

**Description**

`complete_erasure` (`erasure.rs:372-415`) produces `erasure_proof = SHA-256(request_id || subject_id || records_erased)`. A hash of counts is not a commitment to erasure.

**Evidence**

```rust
// crates/kimberlite-compliance/src/erasure.rs:~382 (paraphrased from read)
let erasure_proof = sha256(
    request_id.as_bytes(),
    subject_id.as_bytes(),
    &records_erased.to_le_bytes(),
);
```

A proper erasure proof should commit to:
- The pre-erasure ciphertext digest (Merkle root of stream contents before erasure)
- The post-erasure key-shredding event (Merkle root of destroyed key material)
- A timestamped signature from the server's attestation key

None of these are present. The artifact looks cryptographic but provides zero guarantee that the referenced data is unrecoverable.

**Remediation**

**Priority: P1** (couple to C-1)

1. Redesign the erasure record to include `pre_erasure_ciphertext_root`, `key_shred_digest`, `attestation_sig`.
2. Verify the attestation signature covers the actual effect (key-shredding transaction-id, row-count matching stream length).
3. Publish the proof schema in `docs/concepts/right-to-erasure.md` and add a TLA+ theorem `ErasureProofBindsToAct`.

## Medium Severity Findings

### M-1: Stateless Subscribe handler synthesizes deterministic `subscription_id` from user-controllable inputs

**CWE:** CWE-330 (Use of Insufficiently Random Values), CWE-1188 (Insecure Default Initialization of Resource)

**Risk:** MEDIUM — duplicate code paths with divergent safety properties

**Description**

Two Subscribe handlers exist. The connection-layer path uses a per-connection counter (good). The stateless handler path derives `subscription_id` from `tenant_id * magic + stream_id` with no entropy.

**Evidence**

```rust
// crates/kimberlite-server/src/handler.rs:310-312
let subscription_id = u64::from(request.tenant_id)
    .wrapping_mul(0x517c_c1b7_2722_0a95)
    .wrapping_add(u64::from(req.stream_id));
// No per-connection counter; fully derivable from client-supplied inputs.

// crates/kimberlite-server/src/server.rs:433-439 (the other path)
let sub_id = conn.subscriptions.register(request.tenant_id, req.stream_id, ...);
// per-connection counter (safe)
```

The in-file comment at `server.rs:327-329` says lifecycle requests *"must be handled at the connection layer"* — but the stateless path is still live code that returns `ResponsePayload::Subscribe`. If a future dispatch-order change elevates the stateless path, subscription IDs become cross-connection-predictable.

**Remediation**

**Priority: P2**

1. Delete the `RequestPayload::Subscribe` branch in `handler.rs:304-319` or have it return `ServerError::Replication(...)`. The comment already declares the invariant.
2. If the stateless path must survive for any reason, use a server-side CSPRNG nonce, not derived arithmetic.

### M-2: RBAC / ABAC Kani proofs never model cross-tenant role bindings

**CWE:** CWE-266 (Incorrect Privilege Assignment)

**Risk:** MEDIUM — potential replication of the fixed class of bug in role-binding storage

**Description**

`rbac/kani_proofs.rs` proofs 33-36 and `abac/kani_proofs.rs` proofs 46-49 verify per-policy-object structural invariants (`policy.tenant_id == tenant_a`). No proof constructs two tenants with the same user identifier and verifies that a role granted in tenant A is rejected in tenant B.

**Evidence**

```rust
// crates/kimberlite-rbac/src/kani_proofs.rs:20-78 (sketch)
#[kani::proof]
fn verify_role_separation() {
    let policy_a = kani::any::<Policy>();
    let policy_b = kani::any::<Policy>();
    kani::assume(policy_a.tenant_id != policy_b.tenant_id);

}
```

If role storage were keyed `HashMap<UserId, Role>` instead of `HashMap<(TenantId, UserId), Role>` — the exact failure shape of the old `table_name_index` — these proofs would pass. Needs follow-up to confirm the actual storage keying.

**Remediation**

**Priority: P1** (upgrade to HIGH if follow-up confirms bug presence)

1. Audit `RoleStore` / `PolicyStore` for tenant-keying of the storage map.
2. Add Kani proof `verify_role_binding_per_tenant` analogous to `verify_same_name_succeeds_across_tenants`.
3. Add integration test: grant `u@example.com` Admin in A, attempt Admin-scoped op against B, assert rejection.

### M-3: `trace_alignment::calculate_coverage` is tautological; reports 100% by construction

**Status:** **RESOLVED** by the H-1/H-3 trace-alignment rework. `compute_coverage` at `trace_alignment.rs:494-518` now delegates to an AST validator (`:674-746`) that parses each target file with `syn`, finds the named function, and verifies the declared line range overlaps the function body. Coverage equals AST-verified count, not trace-list length. Two regression tests guard this:

- `test_ast_validator_rejects_injected_bad_range` at `trace_alignment.rs:969` — injects a trace with a `(5_000, 5_100)` range outside the real `apply_committed` body and asserts `ValidationStatus::RangeMismatch`.
- `test_coverage_is_honest_not_tautological` at `trace_alignment.rs:1071` — asserts `theorems_fully_traced == theorems_range_verified` (the AST-verified count) and that `theorems_range_verified > 0`.

No further action required.

**CWE:** CWE-1053 (Missing Documentation for Design), CWE-754 (Improper Check)

**Risk:** MEDIUM — coverage metric used in certification narrative is meaningless

**Description**

The coverage metric is `self.traces.len() / total_tla_theorems_claimed`. Every entry in the `Vec` counts as "fully traced" regardless of whether its cited line range matches any code.

**Evidence**

```rust
// crates/kimberlite-sim/src/trace_alignment.rs:325-343
pub fn calculate_coverage(&self) -> CoverageReport {
    let total = self.traces.len();
    CoverageReport {
        theorems_fully_traced: total,
        total_tla_theorems: total,
        // ... 100% by construction ...
    }
}

// Tests at :478-493 assert 100%.
```

This mechanism allowed `TenantIsolationTheorem` to report 100% coverage while pointing at the wrong lines (the root cause of the April 2026 bug's survival). H-1 is the observable symptom.

**Remediation**

**Priority: P1**

1. Coverage calculation must: (a) parse the target file, (b) locate the named function (`syn` or regex), (c) assert the declared range overlaps the function body, (d) verify the cited VOPR invariant name exists in the invariant registry, (e) verify the cited scenario is registered.
2. Any failed validation downgrades that trace to "claimed but not verified" and reduces the coverage percentage.

### M-6: VSR Byzantine agreement Kani proofs test quorum arithmetic, not agreement under attack

**CWE:** CWE-829 (Inclusion of Functionality from Untrusted Control Sphere)

### M-7: Subquery / CTE RBAC masking not verified to apply recursively

**CWE:** CWE-284 (Improper Access Control)

### M-8: `apply_single_dml_event` silently warns on unknown `type` field

**CWE:** CWE-754 (Improper Check for Unusual or Exceptional Conditions)

**Risk:** MEDIUM — corrupted or malformed DML events become no-ops

**Description**

The projection replay path dispatches on the event's `type` string and falls through on unknown values with a `tracing::warn!`. A corrupted event bearing `"type": "DELETE"` (uppercase) or a mis-encoded type field is silently skipped — the DELETE is not applied, yet the offset advances.

**Evidence**

```rust
// crates/kimberlite/src/kimberlite.rs:611-613
_ => {
    tracing::warn!(?event_type, "unknown DML event type");
}
```

Replay-time safety requires that any malformed event is a process-level failure (the audit-grade invariant: we never silently lose a write). `warn!` + skip is the opposite of that.

**Remediation**

**Priority: P2**

1. Replace the `_ =>` arm with a `KimberliteError::internal` or `panic!` (per `docs/ASSERTIONS.md`: replay-time integrity is compliance-critical).
2. Add a `kimberlite_properties::never!` macro call for the unknown-type branch.
3. Add a test: corrupt an event's `type` field, assert the replay fails loudly.

## Low Severity Findings

### L-1: `apply_masking_to_result` lossy type coercion — every masked value becomes `Value::Text`

**Risk:** LOW — DX and wire compatibility, not safety

**Evidence**


### L-2: Compliance audit `actor: Option<String>` admits anonymous-identity rows

**Evidence**

## SDK Consumer Walks (five user-facing flows)

### Walk 1 — Tenant sealing (H-5 context)

Feature does not exist (see H-5). Walk cannot be completed.

### Walk 2 — Role-scoped SELECT

Trace: `QueryRequest` (wire) → `handler.rs:handle_inner` (auth + tenant isolation, `:161-196`) → `TenantHandle::query_with_policy` (`tenant.rs:1740-1790`) → `RbacFilter::rewrite_statement` (`rbac_filter.rs:112`) → `QueryEngine::query` (parse+plan+execute every call) → `apply_masking_to_result` (`tenant.rs:1801-1843`).

**Holds on happy path**. Role re-validated on every call (no cached prepared-plan reuse over the wire). Column masking applied at result level. Nested subquery rewriting is not exhaustively tested — see M-7.

### Walk 3 — DELETE → erasure (GDPR Art 17)

SQL `DELETE` path **holds**: `Command::Delete` (`kernel.rs:560-617`) → `Effect::UpdateProjection` → `kimberlite.rs:570-610` → `WriteBatch::delete(table_id, pk_key)`. Row is removed from the projection B+tree.

Compliance `ErasureEngine` path **fails**: no actual erasure (C-1, C-4), forgeable ledger, non-cryptographic proof (H-4). The two paths are not wired together — a user calling "erase my data" via the compliance SDK does **not** invoke `Command::Delete` on any stream.

### Walk 4 — Subscription isolation

StreamId encodes tenant in upper 32 bits. Two Subscribe paths exist (M-1); the stateful path is safe, the stateless path has deterministic IDs but is not currently the primary dispatch. Read isolation holds (read_events is tenant-scoped).

### Walk 5 — Audit continuity

Kernel-emitted `Effect::AuditLogAppend` is reliable per commit; stored in `Runtime::audit_log: Vec<_>` (volatile). `ComplianceAuditLog` is also volatile (H-2). No hash chain across events. Restart severs the log.

## Cross-cutting Observations

1. **Pattern: named surface ≠ asserted surface.** Findings C-2, H-1, H-3, M-3, M-6 all share this shape: a name in the test/trace/marketing layer promises property X; the code asserts a strict subset or Y. Fix this systematically with M-3's validator.
2. **Pattern: single-scope proofs.** M-2, M-4, M-5 are all Kani proofs that omit a cross-boundary scenario. Add a standing rule: every per-entity-uniqueness proof needs a corresponding per-tenant (or per-subject) isolation counterpart.
3. **Pattern: compliance ledger ≠ compliance act.** C-1, C-4, H-4, H-2 are all "the product records that X happened" without "X actually happened." Compliance primitives must couple the ledger entry to the cryptographic act, not tolerate arbitrary self-reports.
4. **Pattern: non-durable audit.** H-2 and L-2 both stem from compliance events being process-memory objects. Persistence is a single engineering task that unblocks both.
