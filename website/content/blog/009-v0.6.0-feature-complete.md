Every open-source database eventually faces the same moment. Your formal
methods stack is green. Your consensus is Byzantine-resistant. Your fuzz
nightly passes. Your CI is honest. And then a real consumer тАФ an
application built on your engine тАФ tells you which features are
actually missing.

For Kimberlite, that consumer was a clinic-management application for
Australian healthcare (AU Privacy Act + NDB, GDPR overlap). The team
handed us a gap analysis written against v0.5.1, and it didn't ask for
consensus improvements or new benchmarks or another compliance
framework. It asked for five things:

1. **A schema evolution story we can actually ship migrations against.**
2. **Idempotent upsert тАФ one round-trip, not two.**
3. **Wire-carried GDPR Art 6 consent basis тАФ not a type that's inert on
   the wire.**
4. **Correlated subqueries тАФ the "subjects who have consented to X"
   query shape.**
5. **An in-memory backend for tests that doesn't cost one second per
   suite.**

Plus four more for healthcare completeness: time-travel queries,
column-level masking policies, a one-call subject-erasure helper, and
a queryable audit log.

**v0.6.0 is that list, shipped.** It's not healthcare-specific тАФ
every item generalises to finance ledgers, legal case-management,
government records. Compliance-positioned apps across industries
converge on the same primitives.

## What's new

### SQL surface

**`ON CONFLICT` / UPSERT.** `INSERT INTO t (тАж) VALUES (тАж) ON CONFLICT
(pk) DO UPDATE SET col = EXCLUDED.col` and `ON CONFLICT (pk) DO NOTHING`
now execute as a single atomic kernel command. The `Command::Upsert`
handler emits one event with a `resolution: Inserted | Updated | NoOp`
discriminator тАФ no dual-write window, no two-round-trip helper, no
`afterApply` hook tied to application code. Composes with `RETURNING`.
If you've been carrying an `upsertRow` helper or a
"`UPDATE` then conditional `INSERT`" pattern, every call site collapses
to one `INSERT` with an `ON CONFLICT` clause.

**Correlated subqueries.** The healthcare golden query тАФ "subjects who
have consented to $purpose" тАФ now runs end-to-end:

```sql
SELECT p.* FROM patient_current p
WHERE EXISTS (
  SELECT 1 FROM consent_current c
  WHERE c.subject_id = p.id
    AND c.purpose = 'HealthcareDelivery'
    AND c.withdrawn_at IS NULL
);
```

All four shapes are supported: `EXISTS`, `NOT EXISTS`, `IN (SELECT)`,
`NOT IN (SELECT)`. The planner decorrelates to a semi-join where it
can statically prove equivalence; otherwise a correlated-loop
executor does the obvious thing. A configurable cardinality guard
(`max_correlated_row_evaluations`, default 10M) rejects pathological
outer ├Ч inner cross-products with `CorrelatedCardinalityExceeded`
rather than running out of memory.

**`AS OF TIMESTAMP` / `FOR SYSTEM_TIME AS OF` тАФ runtime wiring
finished.** v0.5.0 shipped the parser plus
`QueryEngine::query_at_timestamp` behind a caller-supplied
timestampтЖТoffset resolver. v0.6.0 ships the default audit-log-backed
resolver so inline SQL works without the caller plumbing one. The SDK
`queryAt()` method now accepts ISO-8601 timestamps and nanosecond
bigints in addition to the existing event-offset form, across all
three SDKs. Ask for a timestamp older than your retention horizon
and you get a clear `AsOfBeforeRetentionHorizon` error тАФ not silence,
not a silently-wrong result.

### Compliance

**Wire protocol v4 тАФ end-to-end `ConsentBasis`.** GDPR Art 6(1)
requires a basis for every consent record тАФ `Consent`, `Contract`,
`LegalObligation`, `VitalInterests`, `PublicTask`,
`LegitimateInterests`. v0.5.0 shipped the `ConsentBasis` SDK type
but the wire didn't carry it, so the native layer hardcoded
`basis: null`. That's worse than not shipping the type at all,
because downstream code branches on a field that's always null.

v0.6.0 bumps `PROTOCOL_VERSION = 3 тЖТ 4` and threads
`Option<GdprArticle>` through `ConsentGrantRequest` and
`ConsentRecord` wire messages, the FFI, the server handler, and all
three SDK `grant()` signatures. The 4-cell compat matrix is tested:

| Client | Server | Expectation                                            |
|--------|--------|--------------------------------------------------------|
| v3     | v4     | Request decodes with `basis = None`; response drops basis |
| v4     | v3     | Clear "server too old" error, not silent data loss     |
| v3     | v3     | Unchanged baseline                                     |
| v4     | v4     | basis round-trips intact through grant тЖТ list          |

**`client.compliance.eraseSubject(subjectId)` тАФ auto-discovery.**
AUDIT-2026-04 C-1 kernel-integrated erasure with a signed
`ErasureCompleted` witness shipped in v0.5.x. v0.6.0 adds
auto-discovery: the SDK now walks every stream tagged `PHI` / `PII`
with a `subject_id` column automatically. The `{ streams: [...] }`
override is still available for cases where you need explicit scope
control. The call is idempotent тАФ a second call returns the existing
signed receipt and emits a "second-call-noop" audit record. A new
VOPR scenario (`EraseSubjectWithCrash`) verifies that an
erase-in-progress crash plus recovery completes with a valid
hash-chain and signed proof.

A hand-rolled erasure orchestrator that walks PHI streams, calls
`deleteRow` in a loop, and hopes nothing crashes mid-way is a class
of code you no longer have to write.

**Audit log тАФ queryable from SDKs.** Kimberlite's audit log has
always been hash-chain tamper-evident. What it wasn't was queryable
from SDKs тАФ so apps maintained parallel `audit_events` streams just
to answer "every access to subject X by actor Y in the last 30 days."
Those parallel streams are no longer necessary:

```typescript
const accesses = await client.compliance.audit.query({
  subjectId: 'patient-123',
  actor: 'practitioner-456',
  fromTs: thirtyDaysAgo,
  limit: 1000,
});
```

The response returns structured rows with
`changedFieldNames: string[]` but **never** `before` / `after`
values тАФ enforced at the `kimberlite-wire` serialisation boundary.
You get shape, not content тАФ which is the only thing a compliance
report needs, and it keeps PHI from leaking through the audit
channel.

**Column-level masking policy CRUD.** The existing `FieldMask`
infrastructure (5 strategies: Redact / Hash / Tokenize / Truncate /
Null, shipped v0.4.x) gets a SQL surface:

```sql
CREATE MASKING POLICY phi_redact AS
  CASE WHEN session_role = 'clinician' THEN @col ELSE '***' END;

ALTER TABLE patient_current
  ALTER COLUMN given_names SET MASKING POLICY phi_redact;
```

Queries hitting `given_names` under a non-clinician role return
`'***'`. Under break-glass, the real value is returned and the
existing break-glass audit record fires. New VOPR scenario
(`MaskingRoleTransition`) proves no unredacted leakage across role
transitions. Composes with RBAC тАФ RBAC filters rows first, masks
apply to surviving columns.

### Testing

**`StorageBackend` trait + `MemoryStorage`.** The concrete `Storage`
struct in `crates/kimberlite-storage` is extracted behind a
`trait StorageBackend`; a new `MemoryStorage` impl does no disk IO,
maintains the hash-chain in memory, and produces deterministic
replays. `Kimberlite::in_memory()` sits alongside
`Kimberlite::open(path)` тАФ no breaking change. The test harness
grows `Backend::InMemory`; TS + Python test modules accept
`{ backend: 'memory' }`:

```typescript
import { createTestKimberlite } from '@kimberlitedb/testing';

const kim = await createTestKimberlite({ backend: 'memory' });
// startup + 1k INSERTs + 1k SELECTs: < 200 ms
```

17.7├Ч faster than the TempDir baseline on Apple M-series in release.
If you've been running a regex-SQL fake in your test suite because
the real kernel was too slow to spin up per-test, the fake is no
longer necessary тАФ you can run the real parser, real planner, real
kernel, and real storage at full speed.

**ALTER TABLE тАФ VOPR crash-recovery scenario + doc-tests.**
`ALTER TABLE ADD / DROP COLUMN` shipped end-to-end in v0.5.0 with
`schema_version` monotonicity assertions. v0.6.0 adds the missing
`AlterTableCrashRecovery` VOPR scenario (`ADD COLUMN` + concurrent
`INSERT`s + storage-crash mid-ALTER тЖТ recovery preserves
`schema_version` monotonicity and event ordering) and the paired
`#[should_panic]` tests on every `schema_version` assertion, per the
assertions-inventory policy.

### Infrastructure

**EPYC nightly campaigns.** Three systemd timer pairs on the Hetzner
EPYC box (`caterwaul`):

- `kimberlite-vopr-nightly` тАФ daily, 19:00 UTC, ~2-4h: VOPR combined
  scenario at 50k iterations.
- `kimberlite-fv-nightly-lite` тАФ daily, 01:00 UTC, ~35m: MIRI +
  Kani-smoke.
- `kimberlite-fv-weekly` тАФ Saturdays, 04:00 UTC: full Alloy + Ivy +
  Coq + Kani with `--unwind 16`.

Twelve new `just` recipes under `vopr-epyc-timer-*` /
`fv-epyc-weekly-timer-*` / `fv-epyc-lite-timer-*` namespaces. Three
false positives caught by the new nightly campaigns are fixed in
v0.6.0.

### Dependency bumps

- **sqlparser 0.54 тЖТ 0.61.** Prerequisite for `ON CONFLICT` (not
  natively representable in 0.54's AST). Sixteen breaking-change
  categories migrated in one dedicated commit; 470 `kimberlite-query`
  tests pass.
- **printpdf 0.7 тЖТ 0.9.** Full rewrite of
  `crates/kimberlite-compliance/src/report.rs` (427 lines) against
  the new layout-tree / Op-based model. Compliance-report rendering

- **Security:** ten RustSec advisories closed via `aws-lc-rs`,
  `rustls-webpki`, and `tar` bumps. See `CHANGELOG.md ┬з [0.6.0]` тЖТ
  `Dependencies` for the list.

## What this unlocks

Before v0.6.0, a compliance-positioned app on Kimberlite typically
carried ~1,000 lines of workaround code per deployment:

| Pattern                                                          | v0.6.0 replacement                                    |
|------------------------------------------------------------------|-------------------------------------------------------|
| Regex-SQL fake or fake-kernel shim for fast tests                | `createTestKimberlite({ backend: 'memory' })`         |
| Custom erasure orchestrator walking PHI streams                  | `client.compliance.eraseSubject(subjectId)`           |
| Parallel `audit_events` stream to answer compliance queries      | `client.compliance.audit.query({тАж})`                  |
| `upsertRow`-style helpers wrapping `UPDATE`-then-`INSERT`        | `INSERT тАж ON CONFLICT (pk) DO UPDATE SET тАж`           |
| `ConsentBasis` hardcoded null because wire couldn't carry it     | Basis round-trips end-to-end through wire v4          |

The pattern is the same across industries. A healthcare app ships
less orchestration code; a finance ledger ships less orchestration
code; a legal case-management app ships less orchestration code. The
primitives are now where they should be тАФ kernel-side, reachable
from every SDK.

## What's next

**v0.7.0** targets the polish that didn't make the v0.6.0 cut: Go
SDK, additional scalar functions (`MOD`, `POWER`, `SUBSTRING`,
`EXTRACT`, `DATE_TRUNC`, `NOW()`) pending the VOPR-sim-vs-wall-clock
threading decision, VOPR workload generators for the v0.6.0 command
families so fault-injection covers the full new surface, and the
topological-order validation pass for the publish pipeline. No
breaking wire changes expected.

**v1.0** is checklist-gated rather than date-gated тАФ see
[`ROADMAP.md`](https://github.com/kimberlitedb/kimberlite/blob/main/ROADMAP.md#v10--checklist-gated)
for the full list. Highlights: third-party SOC 2 Type II audit,
HIPAA attestation with a BAA-capable partner, GDPR readiness review
by independent counsel, Java / C++ / Go SDKs at parity, Coq тЖТ Rust
extraction completed, Ivy тЖТ Apalache migration done, at least one
production deployment running Kimberlite as system of record.
"-compliant" / "-certified" language is earned by audit, not
claimed by design тАФ that baseline was set in v0.4.2 and hasn't
moved.

**Post-v1.0**, Kimberlite Cloud тАФ the managed database service тАФ is
planned. The OSS core stays OSS and remains independently usable
with the same feature set as the cloud layer. Similar model to
CockroachDB Serverless on top of CockroachDB OSS.
