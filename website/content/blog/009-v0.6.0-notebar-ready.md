Every open-source database eventually faces the same moment. Your formal
methods stack is green. Your consensus is Byzantine-resistant. Your fuzz
nightly passes. Your CI is honest. And then a consumer — a real
application built on your engine — tells you which features are actually
missing.

For Kimberlite, that consumer is **Notebar**: a clinic-management app for
Australian healthcare (AU Privacy Act + NDB, with GDPR overlap). Notebar's
authors handed us a gap analysis written against v0.5.1 — and it didn't
ask for consensus improvements, or benchmarks, or another compliance
framework. It asked for five things:

1. **A schema evolution story we can actually ship migrations against.**
2. **Idempotent upsert — one round-trip, not two.**
3. **Wire-carried GDPR Art 6 consent basis — not a type that's inert on
   the wire.**
4. **Correlated subqueries — the "patients who have consented to X" query
   shape.**
5. **An in-memory backend for tests that doesn't cost 1 second per suite.**

Plus four more for healthcare completeness: time-travel queries, column-
level masking policies, a one-call subject-erasure helper, and a queryable
audit log.

**v0.6.0 is that list, shipped.**

---

## What's new

### SQL surface

**`ON CONFLICT` / UPSERT.** `INSERT INTO t (…) VALUES (…) ON CONFLICT (pk)
DO UPDATE SET col = EXCLUDED.col` and `ON CONFLICT (pk) DO NOTHING` now
execute as a single atomic kernel command. The `Command::Upsert` handler
emits one event with a `resolution: Inserted | Updated | NoOp`
discriminator — no dual-write window, no two-round-trip helper, no
`afterApply` hook tied to application code. Composes with `RETURNING`.
Notebar's `upsertRow` helper is deprecated; every call site collapses to
one `INSERT`.

**Correlated subqueries.** The healthcare golden query — "patients who
have consented to HealthcareDelivery" — now runs end-to-end:

```sql
SELECT p.* FROM patient_current p
WHERE EXISTS (
  SELECT 1 FROM consent_current c
  WHERE c.subject_id = p.id
    AND c.purpose = 'HealthcareDelivery'
    AND c.withdrawn_at IS NULL
);
```

All four shapes are supported: `EXISTS`, `NOT EXISTS`, `IN (SELECT)`,
`NOT IN (SELECT)`. The planner decorrelates to a semi-join where it can
statically prove equivalence; otherwise a correlated-loop executor does
the obvious thing. A configurable cardinality guard
(`max_correlated_row_evaluations`, default 10M) rejects pathological
outer × inner cross products with `CorrelatedCardinalityExceeded` rather
than running out of memory.

**`AS OF TIMESTAMP` / `FOR SYSTEM_TIME AS OF` — runtime wiring finished.**
v0.5.0 shipped the parser + `QueryEngine::query_at_timestamp` behind a
caller-supplied timestamp→offset resolver. v0.6.0 ships the default
audit-log-backed resolver so inline SQL works without the caller plumbing
one. The SDK `queryAt()` method now accepts ISO-8601 timestamps and
nanosecond bigints in addition to the existing event-offset form, across
all three SDKs. Ask for a timestamp older than your retention horizon
and you get a clear `AsOfBeforeRetentionHorizon` error — not silence.

### Compliance

**Wire protocol v4 — end-to-end `ConsentBasis`.** GDPR Art 6(1) requires
a basis for every consent record — `Consent`, `Contract`,
`LegalObligation`, `VitalInterests`, `PublicTask`, `LegitimateInterests`.
v0.5.0 shipped the `ConsentBasis` SDK type but the wire didn't carry it,
so `nativeConsentToRecord` hardcoded `basis: null`. That's worse than not
shipping the type at all, because downstream code branches on a field
that's always null.

v0.6.0 bumps `PROTOCOL_VERSION = 3 → 4` and threads `Option<GdprArticle>`
through `ConsentGrantRequest` and `ConsentRecord` wire messages, the Node
FFI, the server handler, and all three SDK `grant()` signatures. The
`AUDIT-2026-04 S4.13` TODO at `compliance.ts:469` is deleted. The 4-cell
compat matrix is tested:

| Client | Server | Expectation |
|---|---|---|
| v3 | v4 | Request decodes with `basis = None`; response drops basis |
| v4 | v3 | Clear "server too old" error, not silent data loss |
| v3 | v3 | Unchanged baseline |
| v4 | v4 | basis round-trips intact through grant → list |

**`client.compliance.eraseSubject(subjectId)` — auto-discovery.**
AUDIT-2026-04 C-1 kernel-integrated erasure with a signed
`ErasureCompleted` witness shipped in v0.5.x. v0.6.0 adds auto-discovery:
the SDK now walks every stream tagged `PHI`/`PII` with a `subject_id`
column automatically. The `{ streams: [...] }` override is still
available. The call is idempotent — a second call returns the existing
signed receipt and emits a "second-call-noop" audit record. A new VOPR
scenario (`EraseSubjectWithCrash`) verifies that an erase-in-progress
crash + recovery completes with a valid hash-chain and signed proof.

Notebar's 100-line `erasure.ts` orchestrator collapses to one line.

**Audit log — queryable from SDKs.** Kimberlite's audit log has always
been hash-chain tamper-evident. What it wasn't was queryable from SDKs —
so Notebar maintained a parallel `audit_events` stream just to answer
"every access to patient X by practitioner Y in the last 30 days." That
stream is deleted in v0.6.0:

```typescript
const accesses = await client.compliance.audit.query({
  subjectId: 'patient-123',
  actor: 'practitioner-456',
  fromTs: thirtyDaysAgo,
  limit: 1000,
});
```

The response returns structured rows with `changedFieldNames: string[]`
but **never** `before`/`after` values — enforced at the
`kimberlite-wire` serialization boundary. You get shape, not content —
which is the only thing a compliance report needs, and it keeps PHI
from leaking through the audit channel.

**Column-level masking policy CRUD.** The existing `FieldMask`
infrastructure (5 strategies: Redact / Hash / Tokenize / Truncate / Null,
shipped v0.4.x) gets a SQL surface:

```sql
CREATE MASKING POLICY phi_redact AS
  CASE WHEN session_role = 'clinician' THEN @col ELSE '***' END;

ALTER TABLE patient_current
  ALTER COLUMN given_names SET MASKING POLICY phi_redact;
```

Queries hitting `given_names` under a non-clinician role return `'***'`.
Under break-glass, the real value is returned and the existing break-
glass audit record fires. New VOPR scenario
(`MaskingRoleTransition`) proves no unredacted leakage across role
transitions.

### Testing

**`StorageBackend` trait + `MemoryStorage`.** The biggest Notebar
drift-elimination lever in v0.6.0. The concrete `Storage` struct in
`crates/kimberlite-storage` is extracted behind a `trait StorageBackend`;
a new `MemoryStorage` impl does no disk IO, maintains the hash-chain in
memory, and produces deterministic replays. `Kimberlite::in_memory()`
sits alongside `Kimberlite::open(path)` — no breaking change. The test
harness grows `Backend::InMemory`; TS + Python test modules accept
`{ backend: 'memory' }`:

```typescript
import { createTestKimberlite } from '@kimberlitedb/testing';

const kim = await createTestKimberlite({ backend: 'memory' });
// startup + 1k INSERTs + 1k SELECTs: < 200 ms
```

The TempDir baseline for the same workload was ~1 second — which is why
Notebar was running ~600 lines of regex SQL parser in
`packages/testing/src/fakes/` instead of using the real kernel. Those
600 lines are deleted in v0.6.0's Notebar migration.

**ALTER TABLE — VOPR crash-recovery scenario + doc-tests.** ALTER TABLE
ADD/DROP COLUMN shipped end-to-end in v0.5.0 with `schema_version`
monotonicity assertions. v0.6.0 adds the missing
`AlterTableCrashRecovery` VOPR scenario (ADD COLUMN + concurrent INSERTs
+ storage-crash mid-ALTER → recovery preserves schema_version
monotonicity and event ordering) and the paired `#[should_panic]` tests
on every `schema_version` assertion, per the assertions-inventory
policy.

### Infrastructure

**EPYC nightly campaigns.** Three systemd timer pairs on the Hetzner
EPYC box (`caterwaul`):

- `kimberlite-vopr-nightly` — daily, 19:00 UTC, ~2-4h: VOPR combined
  scenario at 50k iterations.
- `kimberlite-fv-nightly-lite` — daily, 01:00 UTC, ~35m: MIRI +
  Kani-smoke.
- `kimberlite-fv-weekly` — Saturdays, 04:00 UTC: full Alloy + Ivy +
  Coq + Kani with `--unwind 16`.

Twelve new `just` recipes under `vopr-epyc-timer-*` /
`fv-epyc-weekly-timer-*` / `fv-epyc-lite-timer-*` namespaces. Three
false positives caught by the new nightly campaigns are fixed in
v0.6.0 (see the bug-fixes section of CHANGELOG.md).

### Dependency bumps

- **sqlparser 0.54 → 0.61.** Prerequisite for ON CONFLICT (not
  natively representable in 0.54's AST). Sixteen breaking-change
  categories migrated in one dedicated commit; 470 kimberlite-query
  tests pass.
- **printpdf 0.7 → 0.9.** Full rewrite of
  `crates/kimberlite-compliance/src/report.rs` (427 lines) against the
  new layout-tree / Op-based model. Compliance-report rendering
  contract (SOC 2 / HIPAA PDF artefacts) identical.

## The Notebar migration

v0.6.0 lets Notebar ship a single migration commit that deletes roughly
**1,000 lines of workaround code**:

| Deleted Notebar code | Replaced by |
|---|---|
| `packages/testing/src/fakes/fake-kimberlite.ts` (~350 lines regex SQL) | `createTestKimberlite({ backend: 'memory' })` |
| `packages/testing/src/fakes/sql-parser.ts` (~250 lines) | Same as above — real parser, real kernel |
| `packages/kimberlite-client/src/erasure.ts` (~120 lines orchestrator) | `client.compliance.eraseSubject(subjectId)` |
| `packages/kimberlite-client/src/audit.ts` + `notebar-audit.ts` (~150 lines parallel stream) | `client.compliance.audit.query({...})` |
| `packages/kimberlite-client/src/port.ts::upsertRow` + ~31 call sites | `INSERT ... ON CONFLICT (pk) DO UPDATE SET …` |

Plus the `ConsentBasis` wire now carries the GDPR Art 6 basis that v0.5.0
shipped as a type but never let through the wire.

Notebar's authors have a second vertical example (legal or finance on
the Python SDK) planned against the same surface.

## What's next

**v0.7.0** targets the polish that didn't make the v0.6.0 cut:
multi-statement transactions (if consumers ask for them — the
event-sourced pattern + optimistic concurrency has covered every use
case so far), `DROP TABLE IF EXISTS` + `DELETE FROM t` rowsAffected
fixes, Go SDK, auto-generated traceability matrix from in-source
`AUDIT-2026-NN` markers, and additional scalar functions (`MOD`,
`POWER`, `SUBSTRING`, `EXTRACT`, `DATE_TRUNC`, `NOW()`) pending the
VOPR-sim-vs-wall-clock threading decision.

**v1.0** is production-ready positioning: third-party SOC 2 Type II
audit, HIPAA attestation with a BAA-capable partner, GDPR readiness
review by a qualified DPO, Java / C# / C++ SDKs, Coq → Rust extraction
completed, Ivy migration to Apalache. Only after those ship will
kimberlite.dev's user-facing language flip from "HIPAA-ready /
SOC 2-ready / GDPR-ready" to "HIPAA-compliant / SOC 2-certified /
GDPR-compliant." "-compliant" / "-certified" is a label earned by audit,
not claimed by design — that baseline was set in v0.4.2 and hasn't
moved.
