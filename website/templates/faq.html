Production & readiness

If you're a regulated startup evaluating Kimberlite, start here.

Is Kimberlite production-ready?

Kimberlite is in Developer Preview at v0.8.0. The core is stable: 3,000+ tests, deterministic VOPR simulation, formal verification across TLA+ / Coq / Kani / Ivy / Alloy, and a closed pre-launch security audit. Single-node deployments are stable for prototypes, internal tooling, and compliance research.

For full production deployments at scale, wait for v1.0 — that's when third-party SOC 2 / HIPAA attestations, published performance baselines, multi-node cluster productionisation, and SDK coverage (Go, Java, C++) close. See the ROADMAP v1.0 gates for the full checklist.

Is Kimberlite HIPAA-compliant? SOC 2-certified? FedRAMP-authorized?

Not directly — and we want to be precise about that. Kimberlite ships the cryptographic and architectural primitives that HIPAA, SOC 2, FedRAMP, GDPR, PCI DSS, and ISO 27001 demand: hash-chained immutable audit log, server-walked chain attestation (audit.verifyChain()), AES-256-GCM at rest, Ed25519 signed exports, multi-tenant key isolation, GDPR Article 17 erasure orchestration, GDPR Article 6 consent-basis tracking on the wire, column-level masking policies. Six frameworks are formally modelled in TLA+ with TLAPS proofs in nightly CI.

What we have not done: third-party SOC 2 Type II audit, HIPAA attestation + Business Associate Agreement, FedRAMP authorization letter, GDPR readiness review. Those are v1.0 gates.

What this means for you: a regulated startup can build on Kimberlite today knowing the substrate is sound. The audit letters and certifications are work your team (or we, post-v1.0) still have to do — BAA with your hosting provider, internal controls, pen test, audit firm engagement. Kimberlite enables your compliance posture; certification remains your team's responsibility.

Can I use Kimberlite for my healthcare / fintech / legal / education / government startup today?

Yes — for prototypes, internal tools, compliance research, and learning. The core works, the SQL surface is broad (SELECT with aggregates, joins, CTEs incl. recursive, window functions, time-travel via AS OF TIMESTAMP, upsert via INSERT … ON CONFLICT, JSON operators), and the compliance API surface is reachable from Rust, TypeScript, and Python SDKs.

Wait for v1.0 if you need production multi-node cluster deployment, audited compliance certifications, published performance baselines, API stability guarantees, or commercial support.

See healthcare quickstart and finance quickstart for end-to-end walkthroughs.

What about multi-statement transactions (BEGIN / COMMIT / ROLLBACK)?

Not in v0.8.0. The parser rejects them. Single statements are atomic. Multi-statement transactions are planned post-v1.0 — they intentionally sit behind the v1.0 checklist gates so the immutable-log + derived-view contract stays clean for the initial OSS release.

For cross-table workflows today, design each step to be idempotent and use the outbox pattern (recipe in the cookbook). Kimberlite's append-only model means most "transaction" use cases — debit-then-credit ledger entries, encounter-then-audit insertions — can be expressed as a single batch append against an event stream.

Where are the performance benchmarks?

Not yet published. Hardware re-baseline against current generations is a v0.9.0 target. What we have today: 3,000+ correctness tests, deterministic VOPR simulation that catches consensus bugs, formal verification proofs, and a closed pre-launch security audit. Throughput / latency numbers on reference hardware are coming — we'd rather publish honest numbers than aspirational ones.

Why doesn't SELECT NOW() work?

Known issue in v0.8.0. v0.7.0 shipped the NOW() / CURRENT_TIMESTAMP / CURRENT_DATE sentinel variants in the AST, but the planner-side fold_time_constants pass that substitutes them with the request timestamp before evaluation is a v0.9.0 in-flight item. Until then, the evaluator panics if it sees an unfolded sentinel (this is intentional — we'd rather fail loud than silently return wall-clock time from inside a pure kernel that should be deterministic).

Workaround: pass an explicit timestamp parameter from your application instead of relying on NOW() inside SQL.

Which language SDKs work today?

Rust, TypeScript, and Python are at full parity for the data plane and compliance surface. All three ship connection pooling, real-time subscriptions, typed errors, and the full compliance.{consent, audit, erasure, export, breach} API. See the SDK parity matrix for the per-feature status.

Go Phase 1 (data plane + pool) is a v0.9.0 in-flight item; Phase 2 (compliance) and Phase 3 (typed primitives + framework integrations) follow inside the v0.9.x → v1.0 window. Java and C++ are v1.0 gates.

Single-node or multi-node?

Single-node only for production today. Multi-node Viewstamped Replication consensus is fully implemented (Normal, ViewChange, Recovery, Repair, StateTransfer, Reconfiguration) and PR-gated against TLA+ specs — but the kimberlite-cluster crate that orchestrates a multi-node deployment is marked "not ready for public use." Productionising the cluster operator is a v1.0 gate.
